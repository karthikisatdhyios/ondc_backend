Agentic Payment Settlement — Options & Challenges

Dhiyos · prepared for founder discussion · June 2026

Goal. When the agent books a cart, the platform should pay each merchant the exact amount shown on screen, automatically (off-session), after a one-time user authorization — with the AP2 protocol providing signed proof of intent, cart and payment. The blocker is not AP2 (already built and working); it is the money-movement rail available to an India-registered platform.

The problem has two halves

A · User → Agent (auto-deduction) B · Platform → Merchant (settlement)
Let the user authorize once, then allow the agent to pay on their behalf later, without the user present. Route each merchant's exact share into their own bank account — not the platform's.

Most rails solve only one half well. A production system combines an auto-deduction rail (A) with a split-settlement rail (B).

Where we are today (built & tested)

AP2 + Stripe (per-merchant charges)Working in test
  • One-time card authorization via Stripe SetupIntent (off-session) — the AP2 Payment Mandate.
  • On booking: real, signed AP2 mandates (Intent → per-merchant Cart → Payment) + one off-session Stripe charge per merchant for the exact amount, each verifiable (we confirmed live charges + receipts).
  • Limitation: all funds land in our Stripe balance. Merchants are attributed via metadata + a settlement ledger + signed receipt, but money is not deposited into each merchant's bank.

Options for true settlement (Half B)

1 · Stripe ConnectBlocked for India

The "correct" Stripe answer: connected account per merchant + destination charges, each merchant paid directly.

Challenge: India-registered platforms cannot onboard as Stripe Connect platforms (regulatory/identity verification). Not viable for us.

2 · Per-merchant charges + ledger (current)Demo-grade

Keep Stripe, charge each merchant's exact amount to our account, track attribution in a ledger + AP2 receipt.

Challenge: money never reaches the merchants' banks — fine for a prototype, not for real operations. Currency is USD in test (₹ amounts mirror the numbers).

3 · Razorpay Route / Cashfree Easy SplitIndia-native split

India marketplace rails: one customer payment is split and settled directly to each merchant's bank/UPI; platform deducts commission. Route's "Linked Accounts" map almost 1:1 onto our per-merchant step.

Challenges: each merchant must be a real, KYC'd entity with a bank account (real onboarding); INR-only; requires business verification / sales enablement (not instant self-serve); only the payment-execution step changes — AP2 layer stays.

Options for auto-deduction (Half A)

Card / UPI e-mandateCapped + delayed
  • User authorizes once (OTP/3DS); agent debits later up to a cap.
  • Challenges: per-txn OTP above ~₹15,000; 24-hour pre-debit notification required by RBI before each auto-debit → breaks instant checkout.
PPI wallet (Razorpay Engage)Instant off-session
  • User pre-funds a wallet once; agent spends from balance instantly — no per-txn OTP, no 24h notice.
  • Challenges: see licensing below; KYC limits (min-KYC ≈ ₹10k, full-KYC ≤ ₹2,00,000).

PPI wallet — the licensing fork

Wallet typeCan pay…RBI license?
Closedonly Dhiyos (we are merchant-of-record)None needed
Semi-closedmultiple third-party merchantsFull PPI license (India co., ₹15–25 cr net worth, RBI authorization, audits)
Partner / BaaSthird-party merchants, on a licensed issuer's licenceNo licence for us — ride partner (e.g. Razorpay Engage)

A marketplace paying external merchants is semi-closed → needs a licence or a licensed partner. A closed wallet only works if Dhiyos is the seller-of-record. PPI solves Half A beautifully but the merchant payout (Half B) still needs Route / Easy Split.

Recommendation

Prototype (now, no licence): keep AP2 + a closed-loop wallet (internal balance ledger funded via a Razorpay payment, debited instantly by the agent). Demonstrates the full instant agentic-checkout experience legally.

Production: Razorpay Route for real split settlement to merchant banks (smallest change from our current per-merchant step) + an auto-deduction rail — UPI Autopay / e-mandate for standing orders, or a BaaS PPI wallet (Razorpay Engage) for instant agent spend. AP2 stays unchanged across all of these; only the execution step swaps.

Open decisions for founder: (1) Are we merchant-of-record or a true marketplace? (2) Appetite for KYC onboarding of merchants? (3) Instant agent checkout (needs wallet) vs scheduled orders (e-mandate is fine)?

Limits and RBI rules change periodically — verify current caps and licensing thresholds before committing. Prepared as a discussion aid, not legal/financial advice.
